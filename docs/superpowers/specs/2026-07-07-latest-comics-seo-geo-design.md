# Latest Comics SEO and GEO Publishing Contract

## Goal

Improve the search and generative-search quality of the latest published comics—Audre Lorde, Oliver Sacks, and Django Reinhardt—and prevent future uploads from reverting to generic, weakly citable copy.

## Current State

Production and `main` publish source-backed comic records, canonical reader pages, structured data, a sitemap, AI-crawler rules, and `llms.txt`. The newest comics include a title, subject, date, mortality event, page images, and linked sources. They do not include the richer `citable_summary`, `story_notes`, `page_summaries`, or `sameAs` fields found on several older optimized comics.

The rendering layer therefore falls back to generic summary and story-note templates. Page-image captions also fall back to ordinal labels. This makes the factual text less distinctive, buries the primary answer below the image sequence, weakens image context, and leaves entity resolution incomplete.

## Chosen Approach

Use the latest three comics as a verified backfill while upgrading the publishing contract for every subsequent comic.

This is preferred over a one-time metadata patch because daily publishing would otherwise reproduce the same omissions. It is preferred over new topic landing pages because no current query evidence establishes a specific landing-page opportunity, while the comic-page completeness gap is directly observable in production.

## Content Model

Each newly published or refreshed comic must support:

- `citation_passage`: one self-contained, source-grounded passage of 134–167 words that identifies the subject, explains the mortality event and resulting work, states what the comic covers, and names the source trail without inventing claims.
- `page_summaries`: one concise factual caption for every comic page, in page order.
- `sameAs`: at least one verified entity URL, preferring Wikidata and an authoritative biographical entity page.
- Existing linked `sources`: every source used for factual enrichment must remain a named URL in the comic record.

`citation_passage` is the canonical GEO answer block. Existing `citable_summary` and `story_notes` remain supported for backward compatibility, but the new passage is not generated by concatenating generic fallback sentences.

## Reader Page

Add a visible, server-rendered introduction between the reader toolbar and comic pages:

- Question heading: `Who was {person}, and what is this comic about?`
- The authored `citation_passage` as one paragraph.
- Publication date and compact links to the comic's cited sources.

The introduction remains concise so it does not displace the visual story excessively. It must be present in initial HTML and must not depend on client-side JavaScript, hidden SEO-only text, or an interaction to reveal it.

If a legacy comic has no authored passage, the existing experience remains available; the new introductory block is omitted rather than rendering generic text as an optimized passage.

## Structured Data and AI Guidance

For comics with `citation_passage`:

- Add the passage as `abstract` on the `ComicStory`/`CreativeWork` schema node.
- Add the passage as `abstract` on the `WebPage` schema node while retaining the shorter existing `description` used by search and social metadata.
- Retain source URLs in `citation`, the subject `Person`, entity `sameAs`, image objects, dates, and breadcrumbs.

Update `llms.txt` to describe the latest three issues, each with its canonical URL and authored citation passage. Keep the complete compact comic index below it. This gives answer engines current, high-signal passages without expanding every archive entry into a long block.

## Publishing Workflow

Extend `scripts/add_comic.py` with explicit inputs for the GEO fields. To keep long text and ordered captions reliable, accept a UTF-8 JSON metadata file rather than delimiter-heavy command-line strings. The file may provide `citation_passage`, `page_summaries`, and `sameAs`; core CLI fields and discovered media continue to work as they do now.

When adding or refreshing a comic, the script validates:

- Citation passage word count is between 134 and 167 inclusive.
- Page-summary count exactly matches the number of page images.
- Every entity link and structured source URL is an absolute HTTPS URL.
- At least one entity link and one linked source are present.

Validation applies to the comic being added or refreshed. `--render-only` continues to validate archive-wide baseline metadata without breaking legacy records that have not yet been enriched.

The script writes identical enriched metadata to `comics.json` and `comics/<slug>/comic.json`, preserving their current synchronization contract.

## Error Handling

Publishing fails with actionable messages naming the comic and invalid field when:

- A passage is missing or outside the word-count range.
- Caption count differs from page count.
- A source or entity URL is malformed or non-HTTPS.
- The metadata JSON cannot be read or parsed.

No partial metadata writes occur after validation failure. Existing metadata is preserved when an update fails.
